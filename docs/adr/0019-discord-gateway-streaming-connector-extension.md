# 0019 - Discord Gateway: a Streaming Ingestion Mode and a Connector Lifecycle Extension

- Status: Accepted
- Date: 2026-07-22
- Deciders: Founder/CTO
- Related: [CONNECTOR_SDK.md](../CONNECTOR_SDK.md) Sections 3, 4, [ADR-0017](0017-telegram-sync-and-reconciliation-strategy.md)

## Context

`CONNECTOR_SDK.md` Section 4 defines exactly two ingestion transports - webhook (the provider POSTs to an HTTP endpoint we control) and polling (we pull on an interval) - plus hybrid, a combination of both. Discord's real-time message delivery mechanism is neither: the **Gateway** is a persistent, bidirectional WebSocket connection *we* initiate outbound to Discord, authenticate over (`IDENTIFY`), and then hold open indefinitely, maintaining a heartbeat and handling reconnect/resume, while Discord pushes `Dispatch` events (`MESSAGE_CREATE`, etc.) down it as they happen. There is no HTTP endpoint of ours for Discord to call, and polling for new messages is not how any real Discord bot operates (nobody polls the REST API for new messages - it's Gateway-only for real-time delivery, by Discord's own design and rate-limit posture).

The Sprint 1 `Connector` interface (`packages/connector-sdk/src/connector.ts`) has no notion of a long-lived, provider-initiated connection - every method (`validateCredential`, `initialSync`, `reconcile`, `send`) is request/response-shaped, which fit Telegram's webhook model perfectly (ADR-0017) but does not fit a Gateway connection at all.

`ROADMAP.md`'s own "Notes on Sequencing" anticipated exactly this: "Phases 5-8 exist specifically to pressure-test Phase 4. If any of them require changing the SDK interface, that's expected for Discord (Phase 6) - it's the first real second connector - but should not happen by Slack (Phase 7) or Email (Phase 8)." This is that expected, sanctioned change - not scope creep.

Separately: `CONNECTOR_SDK.md` Section 3.1's `oauth2_redirect` row already correctly names Discord's auth method, and Section 3.2 stage 4 describes proactive OAuth refresh-token handling generically. In practice, Discord's actual ongoing API/Gateway authentication uses a **static, non-expiring bot token** (`Authorization: Bot <token>`), issued once in the Discord Developer Portal - not a per-user, expiring OAuth access token. OAuth2 (`authorization_code` grant, `scope=bot`) is used only at connect time, to let a user pick which of their Discord servers (guilds) to install the bot into and to obtain that guild's id - it is not the credential the connector uses for every subsequent call. No proactive refresh mechanism is needed for Discord's bot-token auth, matching Telegram's shape (ADR-0017) more than a typical user-OAuth flow.

## Decision

**1. `IngestionMode` gains a fourth value: `"streaming"`.** A streaming connector maintains its own persistent, provider-initiated push connection instead of exposing an HTTP endpoint or being polled externally. `defineCapabilityManifest()` treats `"streaming"` the same as `"hybrid"` for Section 4.3's reconciliation requirement - a Gateway connection can still drop/miss events during a disconnect, so a periodic reconciliation pass (fetching recent channel history and diffing, which Discord's REST API genuinely supports, unlike Telegram) remains required, not optional.

**2. The `Connector` interface gains one new, optional method**: `startListening(context: ConnectorContext, onMessage: (raw: unknown) => void): Promise<StreamHandle>`, where `StreamHandle` is `{ stop(): Promise<void> }`. A connector that implements it owns its own connection lifecycle (connect, heartbeat, reconnect-with-backoff, resume) internally and invokes `onMessage` with each raw provider payload as it arrives; the platform (`apps/api`) is responsible for calling `mapMessage()` on what it receives and publishing to the event bus, exactly as it already does for the webhook path - only the *trigger* differs (a callback instead of an HTTP request), not what happens after. This is additive and optional: `MockConnector` and `TelegramConnector` are unaffected, since neither implements it.

**3. Discord's `initialSync`/`reconcile` are implemented for real**, not as a no-op (contrast ADR-0017): Discord's REST API has a genuine channel-history endpoint (`GET /channels/{id}/messages`), so Section 8.1's bounded backfill and Section 4.3's list-and-diff reconciliation are implementable exactly as `CONNECTOR_SDK.md` originally specified. This is the first real test of whether that part of the Sprint 1 design generalizes beyond Telegram's constraints - and it does.

**4. No SDK change for OAuth token refresh.** Discord's bot-token auth doesn't expire, so `CONNECTOR_SDK.md` Section 3.2 stage 4's proactive-refresh behavior isn't triggered by this connector. This is a property of Discord's specific auth shape, not evidence the document's generic OAuth guidance is wrong - a future connector with genuinely expiring user-level OAuth tokens (Gmail, for instance) will need that plumbing, and this ADR deliberately does not build it speculatively ahead of that need.

**5. A minimal WebSocket client dependency (`ws`) is added** to `packages/connector-sdk`. Node's built-in global `WebSocket` client is inconsistent across this project's supported Node range (`engines.node: ">=20.0.0"`); `ws` is the de facto standard, single-purpose, zero-bloat library for this - notably, it is already an indirect dependency of `socket.io` (via `engine.io-client`), which this project already depends on for its own realtime server. This is judged a well-justified, minimal addition, not a new architectural direction.

## Consequences

- `StreamHandle`/`startListening` is a real, permanent addition to the `Connector` interface's surface, even though only Discord uses it today - a future Gateway-shaped provider (e.g. a hypothetical Slack Socket Mode integration) can reuse it without another interface change.
- `apps/api` gains a new kind of platform-side responsibility it didn't have for Telegram: owning a `DiscordGatewayManagerService` that starts a `StreamHandle` per active `LinkedAccount` on boot/connect and stops it on disconnect - a persistent, stateful, in-process resource, unlike the stateless HTTP webhook receiver pattern. This is a real, disclosed increase in `apps/api`'s runtime responsibility, tracked in the phase review as a candidate for the eventual per-connector-worker split `ARCHITECTURE.md`/ADR-0009 already anticipates, not built as one now.
- Certification (`CONNECTOR_SDK.md` Section 16 item 2, "implements the full lifecycle state machine") is unaffected - the streaming connection's health feeds the existing `ConnectorLifecycle` (`active`/`degraded`/`error`) exactly as any other connector's does; `startListening` is additive to, not a replacement for, the lifecycle state machine.
- `Slack` (Phase 7) is explicitly *not* expected to need this extension (`ROADMAP.md`'s own framing) - Slack's Events API is a normal HTTP webhook, closer to Telegram's shape than Discord's. If Phase 7 or 8 turns out to need another SDK interface change, `ROADMAP.md`'s own rule applies: treat it as a signal to stop and reassess, not a routine cost.
